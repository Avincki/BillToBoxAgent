BillToBox Invoice Agent

A supervised, agentic Python system that collects invoices from email, extracts their fields with Claude AI, files the PDFs into Google Drive by accounting quarter, and forwards them to Billtobox with human approval.

Python 3.11 / 3.12 FastAPI Claude AI SQLite + WAL Raspberry Pi

1 Project Overview

BillToBoxAgent is a learning exercise in building a genuinely agentic system — one with a tool-calling loop, policy-based autonomy, and a steering dashboard — rather than a fixed script. It runs as two systemd services on a Raspberry Pi, sitting alongside an existing project called HomeEnergyCenter (HEC) whose conventions it deliberately mirrors.

What it does, end to end

  1. Polls Gmail and Outlook (Doccle is stubbed) for messages with PDF attachments that look like invoices.
  2. Pre-filters candidates with cheap, model-free heuristics so junk never reaches the paid model.
  3. De-duplicates by SHA-256 content hash so the same PDF is never processed twice.
  4. Calls Claude to classify the document and extract structured fields (vendor, date, amount, currency, confidence).
  5. Computes the accounting quarter from the invoice date and files the PDF to Invoices/<year>/<quarter>/ in Google Drive.
  6. Records every action in SQLite as an auditable agent_events trail.
  7. Surfaces everything in a tailnet-only FastAPI dashboard where a human reviews, edits, and approves.
  8. On explicit approval, emails the invoice PDF to the Billtobox upload mailbox via SMTP — never without the approval flag.

Who it serves

A single user (the project owner) doing their own bookkeeping. The dashboard is single-user and tailnet-only — there is no authentication beyond Tailscale network membership.

Core idea: SQLite is the index and audit log; Google Drive is the system of record for the actual PDFs. SQLite stores Drive file IDs, not the bytes.

3 Architecture Overview

Two independent systemd units that share host conventions but not code or virtualenvs:

  • agent-worker — the agent loop, triggered by a systemd timer (daily 07:00). Fetches, extracts, files. Type=oneshot — runs one cycle and exits.
  • dashboard — a long-lived FastAPI app (Type=simple) bound to 127.0.0.1:9003, exposed over the tailnet via Caddy at public :8003.

Data flow per run

# For each enabled source (gmail, outlook):
fetch since watermark
  → prefilter        # cheap, model-free gate
  → check_duplicate  # SHA-256 content hash
  → extract_invoice  # Claude — the only paid call
  → confidence gate  # >= 0.85 auto-approve, else flag_for_review
  → ensure_quarter_folder + store_pdf_to_drive
  → advance watermark
close runs row with counts/errors
# every step writes a redacted agent_events row

Why WAL

Because the worker and dashboard are two separate processes touching the same SQLite file, the database runs in WAL (Write-Ahead Logging) mode. The worker is the sole writer; the dashboard mostly reads and writes only small status fields.

The two-layer observability design (a deliberate owner priority):
  • Layer A — structlog JSON file + /api/logs/stream SSE live tail + /logs//debug views.
  • Layer B — the agent_events SQLite table recording every tool call/result/decision (secrets & PDF bytes redacted), shown as an "Agent activity" timeline and a per-invoice audit trail, and exposed to Claude itself as a read-only get_agent_events tool so it can self-correct.

4 Key Concepts

TermWhat it means
content_hashSHA-256 of the PDF bytes, stored with a UNIQUE constraint. The idempotency key: a PDF already seen is never reprocessed or re-uploaded.
WatermarkPer-source last-seen message timestamp stored in source_status. Reruns only fetch messages newer than the watermark.
period_for()The pure, I/O-free function mapping an invoice date to (fy_label, quarter). Drives the Drive folder layout.
Confidence gateClaude returns a confidence 0.0–1.0; items below the configured threshold (0.85) are not auto-approved — they go to flag_for_review.
PrefilterModel-free accept/reject gate: PDF magic bytes, sender allow/blocklist, subject keywords. Runs before any paid Claude call.
agent_eventsThe audit log table — every tool call, result, and decision, redacted of secrets and PDF bytes.
redact()Utility at the agent_events write boundary: scrubs API keys/tokens, replaces PDF bytes with hash + length.
Human-approval flagAn invoice status (upload_approved) set only via the dashboard. The worker never emails Billtobox without it.

The accounting-quarter rules

period_for(invoice_date, fy_start_month=1, fy_label_prefix="")
# Calendar default (fy_start_month=1):
#   Jan–Mar → Q1   Apr–Jun → Q2   Jul–Sep → Q3   Oct–Dec → Q4
# Offset fiscal year (e.g. fy_start_month=4):
#   Apr–Jun → Q1 ... Jan–Mar → Q4 of the PRIOR FY label
# The FY is labelled by its START year.

The 10 agent tools

In Phase 4 the linear pipeline becomes a tool-calling loop registering these:

  • search_mail, get_pdf, extract_invoice, check_duplicate
  • ensure_quarter_folder, store_pdf_to_drive
  • queue_billtobox_upload, email_to_billtobox, flag_for_review
  • get_agent_events — read-only; lets Claude inspect its own history and self-correct (the 10th, added during planning)

Note the split: pipeline/ holds linear-worker steps that aren't owned by an external-service domain. agent/ is reserved for the future LLM tool-loop. This distinction was introduced deliberately in task 12.

6 Setup Prerequisites

Local development

  1. Install Python 3.12 (or 3.11). Verify with python --version.
  2. Create and activate a virtual environment, then pip install -e ".[dev]".
  3. Copy config.example.yaml → config.yaml and fill placeholders.
  4. Run the full toolchain to confirm a clean tree:
ruff check . && black --check . && mypy src && pytest
